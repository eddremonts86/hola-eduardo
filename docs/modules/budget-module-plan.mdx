## 1. Visión General

Crear un módulo `budgets` que se **integra profundamente con el módulo `transactions` existente** en lugar de duplicarlo. Los budgets son contenedores financieros multi-scope (personal, proyecto, departamento, empresa) que agregan transacciones vinculadas. La tabla `transactions` existente evoluciona para soportar tanto uso empresarial como personal. Integración completa con el sistema de IA existente (RAG, chat, action cards) para análisis financiero inteligente.

### Objetivos Principales

- Centralizar ingresos y gastos manuales dentro de presupuestos con alcance definido
- Permitir categorización flexible con límites de gasto por categoría
- Gestionar transacciones recurrentes (mensual, trimestral, semestral, anual)
- Detectar y alertar cuando un presupuesto ha sido excedido (over-budget)
- Soporte multiusuario (familias, equipos, empresas)
- Integración con IA para análisis, predicciones y creación por voz/chat

### Objetivos Avanzados (post-MVP)

- Soporte multi-divisa con conversión
- Integración bancaria y CSV import
- Predicciones ML avanzadas
- Templates y clonación de presupuestos

## 2. Decisiones de Arquitectura

### D1: Extender `transactions`, no duplicar

La tabla `transactions` existente gana: `budgetId` (FK), `isPrivate` (boolean), y `customerName`/`customerEmail` se vuelven **nullable** (las transacciones personales no necesitan un cliente). Todos los flujos existentes siguen funcionando — las transacciones enterprise operan igual que antes.

**Razonamiento**: La tabla `transactions` ya tiene la infraestructura de montos, fechas, categorías, aprobaciones. Duplicar sería crear una isla de datos sin relación con el sistema financiero existente. Al extender con un FK opcional, una transacción puede pertenecer a un budget sin romper nada.

### D2: Budgets son contenedores con scope

Un budget tiene un enum `scope` (`personal | project | department | company`) con FKs nullable explícitas (`projectId`, `departmentId`) — sin FK polimórfico (anti-pattern). La combinación `scope` + FK determina el contexto.

**Razonamiento**: Permite vincular presupuestos directamente a entidades ya existentes (projects, departments) sin crear jerarquías complejas. Cada scope es independiente: un presupuesto de proyecto agrupa las transacciones del proyecto; uno personal, las del usuario.

### D3: Reutilizar módulo `categories` existente

Los **límites de gasto por categoría** referencian la tabla `categories` existente vía una junction table `budget_category_limits`. No se crea una tabla `budget_categories` separada.

**Razonamiento**: Las categorías existentes (Food, Transport, Housing, etc.) son perfectamente reutilizables para presupuestos. Crear categorías nuevas sería duplicar datos y romper el principio plug&play. La junction table permite asignar montos máximos a cada categoría dentro de un presupuesto específico.

### D4: Múltiples contribuidores vía `budget_members`

Cualquier usuario puede ser añadido a un budget con un rol (`admin | contributor | viewer`). Admin = control total, contributor = agregar/editar transacciones propias, viewer = solo lectura. El owner del budget es siempre admin implícito.

**Razonamiento**: Soporta el caso de uso de familias compartiendo un presupuesto doméstico, equipos con presupuesto de proyecto, o departamentos con presupuesto asignado. Los roles limitan quién puede hacer qué sin complejidad adicional.

### D5: Integración con IA vía pipeline RAG existente

Budget intents, action cards, y RAG indexing se conectan a los puntos de extensión del módulo AI existente — no se necesita infraestructura nueva de IA.

**Razonamiento**: El sistema AI ya tiene: detección de intents por keywords (EN/ES), detección de acciones CRUD, action confirmation cards, y RAG via ChromaDB. Solo hay que añadir entries para `budget`/`presupuesto` en cada punto de extensión.

### D6: Montos en centavos (integer)

Consistente con los patrones existentes de `transactions.amount` y `projects.budget`.

**Razonamiento**: Evita problemas de punto flotante. $45.50 se almacena como `4550`. El formateo a display se hace con `Intl.NumberFormat` en el cliente.

## 3. Modelo de Datos

### 3.1 Enums Nuevos

```sql
-- Alcance del presupuesto
CREATE TYPE budget_scope AS ENUM ('personal', 'project', 'department', 'company');

-- Estado del presupuesto
CREATE TYPE budget_status AS ENUM ('active', 'closed', 'archived');

-- Tipo de período
CREATE TYPE budget_period_type AS ENUM ('monthly', 'quarterly', 'semiannual', 'annual', 'one_time');

-- Rol de miembro de presupuesto
CREATE TYPE budget_member_role AS ENUM ('admin', 'contributor', 'viewer');

-- Frecuencia de recurrencia
CREATE TYPE budget_recurrence_frequency AS ENUM ('monthly', 'quarterly', 'semiannual', 'annual');

-- Estado de regla de recurrencia
CREATE TYPE budget_recurrence_status AS ENUM ('active', 'paused', 'completed');
```

### 3.2 Modificaciones a Tabla Existente: `transactions`

| Cambio          | Antes           | Después                                   | Motivo                                          |
| --------------- | --------------- | ----------------------------------------- | ----------------------------------------------- |
| `customerName`  | `text NOT NULL` | `text` (nullable)                         | Transacciones personales no tienen cliente      |
| `customerEmail` | `text NOT NULL` | `text` (nullable)                         | Idem                                            |
| `budgetId`      | —               | `text FK → budgets.id ON DELETE SET NULL` | Vincular transacción a un presupuesto           |
| `isPrivate`     | —               | `boolean DEFAULT false`                   | Transacciones privadas en grupo no son visibles |
| `createdAt`     | —               | `timestamp DEFAULT NOW()`                 | Auditoría                                       |
| `updatedAt`     | —               | `timestamp DEFAULT NOW()`                 | Auditoría                                       |

**Impacto en código existente**: Las transacciones enterprise siguen funcionando porque `customerName`/`customerEmail` eran los únicos NOT NULL que cambian. El código de approval (`canApproveTransaction`, `approvedBy`, `approvedAt`) no se toca. Los nuevos campos (`budgetId`, `isPrivate`) son opcionales.

### Fase 9: Motor de Recurrencia

**Depende de: Fase 4.**

#### 9A. `processRecurrenceRules()` server function

**Archivo**: `src/modules/budgets/api/budget-recurrences.fn.ts`

```ts
export async function processRecurrenceRules(budgetId?: string): Promise<number> {
  const db = await loadDb()
  const now = new Date()

  // Query due rules
  let query = db
    .select()
    .from(budgetRecurrenceRules)
    .where(
      and(eq(budgetRecurrenceRules.status, 'active'), lte(budgetRecurrenceRules.nextDate, now)),
    )

  if (budgetId) {
    query = query.where(eq(budgetRecurrenceRules.budgetId, budgetId))
  }

  const dueRules = await query
  let processedCount = 0

  for (const rule of dueRules) {
    let nextDate = new Date(rule.nextDate)

    // Loop: handle missed periods
    while (nextDate <= now) {
      // 1. Create transaction
      await db.insert(transactions).values({
        id: crypto.randomUUID(),
        amount: rule.amount,
        date: nextDate,
        categoryId: rule.categoryId,
        budgetId: rule.budgetId,
        userId: rule.userId,
        description: `[Auto] ${rule.description || 'Recurring transaction'}`,
        isPrivate: false,
        createdAt: now,
        updatedAt: now,
      })

      // 2. Advance nextDate
      switch (rule.frequency) {
        case 'monthly':
          nextDate.setMonth(nextDate.getMonth() + 1)
          break
        case 'quarterly':
          nextDate.setMonth(nextDate.getMonth() + 3)
          break
        case 'semiannual':
          nextDate.setMonth(nextDate.getMonth() + 6)
          break
        case 'annual':
          nextDate.setFullYear(nextDate.getFullYear() + 1)
          break
      }

      processedCount++
    }

    // 3. Update rule with new nextDate
    await db
      .update(budgetRecurrenceRules)
      .set({ nextDate, updatedAt: now })
      .where(eq(budgetRecurrenceRules.id, rule.id))
  }

  return processedCount
}
```

```ts
  const start = new Date(budget.startDate)

  switch (budget.periodType) {
    case 'monthly': {
      const periodStart = new Date(now.getFullYear(), now.getMonth(), 1)
      const periodEnd = new Date(now.getFullYear(), now.getMonth() + 1, 0, 23, 59, 59)
      return { periodStart, periodEnd }
    }
    case 'quarterly': {
      // Align to budget start date
      const monthsSinceStart =
        (now.getFullYear() - start.getFullYear()) * 12 + now.getMonth() - start.getMonth()
      const currentQuarter = Math.floor(monthsSinceStart / 3)
      const periodStart = new Date(start)
      periodStart.setMonth(start.getMonth() + currentQuarter * 3)
      const periodEnd = new Date(periodStart)
      periodEnd.setMonth(periodEnd.getMonth() + 3)
      periodEnd.setDate(periodEnd.getDate() - 1)
      return { periodStart, periodEnd }
    }
    case 'semiannual': {
      // Same pattern, offset 6 months
      const monthsSinceStart =
        (now.getFullYear() - start.getFullYear()) * 12 + now.getMonth() - start.getMonth()
      const currentHalf = Math.floor(monthsSinceStart / 6)
      const periodStart = new Date(start)
      periodStart.setMonth(start.getMonth() + currentHalf * 6)
      const periodEnd = new Date(periodStart)
      periodEnd.setMonth(periodEnd.getMonth() + 6)
      periodEnd.setDate(periodEnd.getDate() - 1)
      return { periodStart, periodEnd }
    }
    case 'annual': {
      const yearsSinceStart = now.getFullYear() - start.getFullYear()
      const periodStart = new Date(start)
      periodStart.setFullYear(start.getFullYear() + yearsSinceStart)
      if (periodStart > now) periodStart.setFullYear(periodStart.getFullYear() - 1)
      const periodEnd = new Date(periodStart)
      periodEnd.setFullYear(periodEnd.getFullYear() + 1)
      periodEnd.setDate(periodEnd.getDate() - 1)
      return { periodStart, periodEnd }
    }
    case 'one_time': {
      return {
        periodStart: start,
        periodEnd: budget.endDate ? new Date(budget.endDate) : new Date('2099-12-31'),
      }
    }
  }
}
```

### 6.3 Motor de Recurrencia

```
┌───────────────────────────────────────────────────────────┐
│               Trigger Points                               │
├───────────────────────────────────────────────────────────┤
│                                                           │
│  1. Page load (lazy)        2. API endpoint (cron)        │
│  getBudgetByIdFn()          POST /api/budgets/            │
│  → processes rules for      process-recurrences           │
│    THIS budget only         → processes ALL due rules     │
│                                                           │
└───────────────────────┬───────────────────────────────────┘
                        │
         ┌──────────────▼───────────────┐
         │ SELECT budget_recurrence_rules│
         │ WHERE status = 'active'       │
         │ AND next_date <= NOW()        │
         └──────────────┬───────────────┘
                        │
         ┌──────────────▼───────────────┐
         │ For each rule:                │
         │                               │
         │ LOOP while nextDate <= today: │
         │                               │
         │  1. INSERT transaction {      │
         │     amount: rule.amount,      │
         │     budgetId: rule.budgetId,  │
         │     categoryId: rule.catId,   │
         │     userId: rule.userId,      │
         │     date: nextDate,           │
         │     description: "[Auto]..."  │
         │  }                            │
         │                               │
         │  2. Advance nextDate:         │
         │     monthly    → +1 month     │
         │     quarterly  → +3 months    │
         │     semiannual → +6 months    │
         │     annual     → +1 year      │
         │                               │
         │ UPDATE rule SET               │
         │   next_date = advancedDate    │
         └──────────────────────────────┘

  Handles missed periods:
  If cron didn't run for 3 months on a monthly rule,
  the loop creates 3 transactions (one per missed month).
```

## 10. Consideraciones Adicionales

### ¿Jerarquía de budgets?

Actualmente cada budget es independiente (flat). Si se necesita que un company budget agrupe automáticamente los department budgets (roll-up), requeriría un `parentBudgetId` FK. **Recomendación**: dejar flat en MVP, agregar hierarchy en Phase 2.

### ¿Una transacción en múltiples budgets?

Con FK simple (`budgetId`), una transacción → un budget. Si necesitas many-to-many (ej: la misma factura afecta al budget del proyecto Y al del departamento), necesitaría una junction table `budget_transactions`. **Recomendación**: single budget en MVP, evaluar en Phase 2 si surge la necesidad.

```ts
const formatter = new Intl.NumberFormat(locale, { style: 'currency', currency: budget.currency })
```

#### P0-2: Sin atomicidad SQL — operaciones multi-paso sin transacción de BD

**Problema**: Al crear un budget (INSERT budget → INSERT budget_member), si el segundo INSERT falla, el budget queda huérfano sin owner. En recurrencias (INSERT transaction → UPDATE rule.nextDate), si la actualización falla, la misma regla se reprocessa en el próximo ciclo creando transacciones duplicadas.

**Impacto**: Datos financieros inconsistentes, transacciones perdidas o duplicadas.

**Mitigación**: Envolver en `db.transaction()` TODAS las operaciones multi-paso:

```ts
// budgets.fn.ts
export const createBudgetFn = createServerFn({ method: 'POST' }).handler(async ({ data }) => {
  const user = await requireCurrentAppUser()
  const db = await loadDb()

  return await db.transaction(async (tx) => {
    const [budget] = await tx
      .insert(budgets)
      .values({
        ...data,
        ownerId: user.id,
      })
      .returning()

    await tx.insert(budgetMembers).values({
      budgetId: budget.id,
      userId: user.id,
      role: 'admin',
    })

    return budget
  })
})

// budget-recurrences.fn.ts — dentro del loop de procesamiento
await db.transaction(async (tx) => {
  const [created] = await tx.insert(transactions).values(newTransaction).returning()
  await tx
    .update(budgetRecurrenceRules)
    .set({ nextDate: computeNextDate(rule), lastRunAt: new Date() })
    .where(eq(budgetRecurrenceRules.id, rule.id))
  return created
})
```

**Checklist**: `createBudgetFn` usa `db.transaction()` ✓ · recurrence processor usa `db.transaction()` por regla ✓ · cualquier operación con ≥2 writes usa `db.transaction()` ✓

#### P0-3: Race condition en el chequeo over-budget

**Problema**: Dos usuarios del mismo group budget crean transacciones simultáneamente. Ambas requests leen el health status antes de que la otra escriba. Ambas pasan el check. El budget termina $X por encima del límite sin ninguna advertencia.

**Impacto**: Pérdida silenciosa de control presupuestario. En un deployment con varios usuarios el problema es garantizado.

**Mitigación**: Usar `SELECT FOR UPDATE` para serializar el acceso por budget:

```ts
// budget-analytics.fn.ts — dentro de db.transaction()
const [budgetRow] = await tx
  .select()
  .from(budgets)
  .where(eq(budgets.id, budgetId))
  .for('update') // Drizzle: .for('update') → FOR UPDATE en Postgres
  // Alternativa: pg_try_advisory_lock si se prefiere no bloquear la fila

const { spent } = await tx
  .select({ spent: sql<number>`SUM(${transactions.amount})` })
  .from(transactions)
  .where(eq(transactions.budgetId, budgetId))
  .then(r => r[0])

const health = computeHealth(budgetRow, spent)
// INSERT transaction dentro de la misma transacción
await tx.insert(transactions).values(...)
return health
```

**Checklist**: `createTransactionFn` con budgetId usa `SELECT FOR UPDATE` ✓ · health check y INSERT están en el mismo `db.transaction()` ✓

#### P0-4: Recurrencias — doble procesamiento por cargas concurrentes

**Problema**: Dos pestañas abiertas del mismo budget, o dos usuarios del mismo group budget, cargan la página simultáneamente. `processRecurrenceRules()` es invocado en ambas. Ambas leen las reglas con `status='active' AND nextDate <= now` antes de que cualquiera actualice `nextDate`. Resultado: transacciones duplicadas.

**Impacto**: Gastos fantasma que inflan el consumo del budget, posibles pagos dobles si el sistema se conecta a ERP.

**Mitigación**: Advisory lock de PostgreSQL por budget:

```ts
// budget-recurrences.fn.ts
async function processRecurrenceRulesForBudget(budgetId: string) {
  const db = await loadDb()

  // Convertir budgetId (UUID) a bigint para pg_try_advisory_lock
  const lockKey = BigInt('0x' + budgetId.replace(/-/g, '').slice(0, 16))

  return await db.transaction(async (tx) => {
    // Si no se obtiene el lock, otro proceso está procesando este budget
    const [{ locked }] = await tx.execute(
      sql`SELECT pg_try_advisory_xact_lock(${lockKey}) AS locked`,
    )
    if (!locked) return { skipped: true }

    const dueRules = await tx
      .select()
      .from(budgetRecurrenceRules)
      .where(
        and(
          eq(budgetRecurrenceRules.budgetId, budgetId),
          eq(budgetRecurrenceRules.status, 'active'),
          lte(budgetRecurrenceRules.nextDate, new Date()),
        ),
      )

    for (const rule of dueRules) {
      await tx.insert(transactions).values(buildTransaction(rule))
      await tx
        .update(budgetRecurrenceRules)
        .set({ nextDate: computeNextDate(rule), lastRunAt: new Date() })
        .where(eq(budgetRecurrenceRules.id, rule.id))
    }

    return { processed: dueRules.length }
  })
}
```

**Checklist**: `processRecurrenceRulesForBudget` usa advisory lock ✓ · lock scope es por budgetId (no global) ✓ · test con 2 requests concurrentes retorna `{ skipped: true }` correctamente ✓

#### P0-6: Explosión de catch-up en recurrencias — regla antigua genera N transacciones sin límite

**Problema**: Una regla mensual con `nextDate` hace 5 años genera 60 transacciones en una sola invocación de `processRecurrenceRules`. Sin límite, esto puede crear miles de transacciones en segundos, saturar la BD y hacer el budget completamente inútil.

**Impacto**: DoS de facto en el módulo de budgets, datos históricos falsos, posible timeout de la request.

**Mitigación**:

```ts
const MAX_CATCHUP_ITERATIONS = 24 // máx 2 años de catch-up para recurrencia mensual

async function processSingleRule(rule: BudgetRecurrenceRule, tx: DbTransaction) {
  let iterations = 0
  let nextDate = rule.nextDate

  while (nextDate <= new Date() && iterations < MAX_CATCHUP_ITERATIONS) {
    await tx.insert(transactions).values(buildTransaction(rule, nextDate))
    nextDate = computeNextDate(rule, nextDate)
    iterations++
  }

  if (iterations >= MAX_CATCHUP_ITERATIONS) {
    console.warn(`[Recurrence] Rule ${rule.id} hit catch-up limit. Pausing.`)
    await tx
      .update(budgetRecurrenceRules)
      .set({ status: 'paused', pausedReason: 'catch-up-limit' })
      .where(eq(budgetRecurrenceRules.id, rule.id))
  } else {
    await tx
      .update(budgetRecurrenceRules)
      .set({ nextDate, lastRunAt: new Date() })
      .where(eq(budgetRecurrenceRules.id, rule.id))
  }
}
```

**Checklist**: `MAX_CATCHUP_ITERATIONS = 24` definido como constante ✓ · regla se pausa automáticamente si se excede el límite ✓ · warning en logs con `rule.id` ✓ · UI muestra `pausedReason: 'catch-up-limit'` con mensaje explicativo ✓

#### P1-4: Sin enforcement de timezone — cálculos de período incorrectos

**Problema**: `new Date()` en Node.js usa el timezone del servidor. Si el servidor está en UTC+2, `new Date()` a las 23:00 local retorna el día siguiente en UTC. Los cálculos de período (`getPeriodBounds()`) pueden asignar transacciones al período incorrecto.

**Impacto**: Transacciones aparecen en el período equivocado. Over-budget detection puede fallar o dispararse incorrectamente.

**Mitigación**:

```ts
// model/period-utils.ts — SIEMPRE trabajar en UTC
export function getPeriodBounds(
  periodType: BudgetPeriodType,
  referenceDate: Date = new Date(), // Date en JS siempre es UTC internamente
): { start: Date; end: Date } {
  // Usar UTC methods, no locales
  const now = new Date(Date.now()) // explícitamente UTC epoch
  const year = now.getUTCFullYear()
  const month = now.getUTCMonth()
  const day = now.getUTCDate()

  switch (periodType) {
    case 'monthly':
      return {
        start: new Date(Date.UTC(year, month, 1, 0, 0, 0, 0)),
        end: new Date(Date.UTC(year, month + 1, 0, 23, 59, 59, 999)),
      }
    case 'quarterly': {
      const quarter = Math.floor(month / 3)
      return {
        start: new Date(Date.UTC(year, quarter * 3, 1)),
        end: new Date(Date.UTC(year, quarter * 3 + 3, 0, 23, 59, 59, 999)),
      }
    }
    // ...
  }
}

// Recurrencias — nextDate siempre en UTC
function computeNextDate(rule: BudgetRecurrenceRule, from: Date): Date {
  const d = new Date(from)
  switch (rule.frequency) {
    case 'monthly':
      return addMonths(d, rule.interval) // date-fns (ver P2-3)
    case 'weekly':
      return addWeeks(d, rule.interval)
    case 'yearly':
      return addYears(d, rule.interval)
  }
}
```

**Checklist**: `getPeriodBounds` usa `Date.UTC()` exclusivamente ✓ · no hay `new Date().getMonth()` (local) en period-utils ✓ · `nextDate` de recurrencias usa `addMonths` de date-fns ✓

**Problema**: El budget tiene `currency: 'USD'`. Una transacción de `currency: 'EUR'` con `budgetId` apunta al mismo budget. El sistema suma EUR + USD como si fueran iguales. Los totales son incorrectos.

```ts
  if (data.currency && data.currency !== budget.currency) {
    throw new Error(
      `Currency mismatch: transaction is ${data.currency} but budget is ${budget.currency}`,
    )
  }

  // Si la transacción no especifica currency, heredarla del budget
  data.currency = budget.currency
}
```

**Checklist**: `createTransactionFn` valida currency match ✓ · `updateTransactionFn` también valida si se cambia `budgetId` o `currency` ✓ · error descriptivo retornado al cliente ✓

#### P1-6: Budget mutable después de transacciones existentes — historial invalidado

**Problema**: Cambiar `periodType` de `monthly` a `quarterly` en un budget con 3 meses de transacciones hace que todas las transacciones históricas aparezcan asignadas a períodos diferentes. El historial queda sin sentido.

```ts
// budgets.fn.ts — updateBudgetFn
```

#### P1-7: Suma de límites por categoría puede exceder el target del budget

**Problema**: Si un budget tiene `targetAmount = 10,000`, se pueden crear 10 límites de categoría de $1,500 cada uno (total $15,000). El sistema mostrará que se han consumido $15,000 de un budget de $10,000 — sin ningún error.

**Impacto**: Los límites de categoría dejan de ser útiles como herramienta de control.

**Mitigación**:

```ts
// budget-limits.fn.ts
export const upsertBudgetLimitFn = createServerFn({ method: 'POST' }).handler(async ({ data }) => {
  const db = await loadDb()

  // Calcular total de límites actuales (excluyendo el que se está editando)
  const [{ currentTotal }] = await db
    .select({
      currentTotal: sql<number>`COALESCE(SUM(${budgetCategoryLimits.allocatedAmount}), 0)`,

        eq(budgetCategoryLimits.budgetId, data.budgetId),
        data.id ? ne(budgetCategoryLimits.id, data.id) : undefined,

    .select({ targetAmount: budgets.targetAmount })

        `Available: ${targetAmount - currentTotal}. Requested: ${data.allocatedAmount}`,
    )
  }

  // upsert...
})
```

### P2 — Problemas de Severidad Media (6)

#### P2-1: Sin rate limiting en endpoints financieros

**Problema**: No existe ningún middleware de rate limiting. Un usuario (o bot) puede crear 10,000 transacciones por segundo, saturando la BD y generando datos de analytics inútiles.

**Mitigación**: Rate limiting por `userId` en server functions críticos:

```ts
// shared/lib/rate-limit.ts (nuevo archivo)
const RateLimiter = new Map<string, { count: number; resetAt: number }>()

export function checkRateLimit(key: string, maxPerMinute = 60) {
  const now = Date.now()
  const entry = RateLimiter.get(key) ?? { count: 0, resetAt: now + 60_000 }

  if (now > entry.resetAt) {
    entry.count = 0
    entry.resetAt = now + 60_000
  }

  entry.count++
  RateLimiter.set(key, entry)

  if (entry.count > maxPerMinute) {
    throw new Error(`Rate limit exceeded. Try again in ${Math.ceil((entry.resetAt - now) / 1000)}s`)
  }
}

// En createTransactionFn:
const user = await requireCurrentAppUser()
checkRateLimit(`transaction:create:${user.id}`, 60) // max 60 transacciones/min
```

**Nota**: Para producción real, usar Redis-backed rate limiting o un middleware a nivel de reverse proxy (nginx, Cloudflare). El ejemplo de Map en memoria no persiste entre reinicios del servidor.

**Checklist**: `createTransactionFn` tiene rate limiting por userId ✓ · `processRecurrencesFn` tiene rate limiting global ✓

#### P2-2: Sin log de auditoría inmutable

**Problema**: No hay registro de quién hizo qué y cuándo: quién creó un budget, quién modificó el target, quién eliminó una transacción. En contextos corporativos esto es un requisito de compliance básico.

**Mitigación**: Tabla append-only de audit log:

```ts
// schema.ts — nueva tabla
export const budgetAuditLog = pgTable('budget_audit_log', {
  id: uuid('id').primaryKey().defaultRandom(),
  budgetId: uuid('budget_id').references(() => budgets.id, { onDelete: 'restrict' }), // no borrar si hay audit
  userId: text('user_id').notNull(),
  action: text('action').notNull(), // 'create_budget' | 'update_budget' | 'delete_transaction' | ...
  entityId: text('entity_id'), // ID del objeto afectado
  payload: jsonb('payload'), // diff del cambio (before/after)
  createdAt: timestamp('created_at').defaultNow().notNull(),
  // NO hay deletedAt — esta tabla es immutable
})

// Helper para loggear:
async function auditLog(
  tx: DbTransaction,
  action: string,
  { budgetId, userId, entityId, payload }: AuditLogEntry,
) {
  await tx.insert(budgetAuditLog).values({ budgetId, userId, action, entityId, payload })
}

// Uso en createBudgetFn dentro de db.transaction():
await auditLog(tx, 'create_budget', { budgetId: budget.id, userId: user.id, payload: data })
```

**Checklist**: `budgetAuditLog` tabla existe en schema ✓ · `createBudgetFn` loggea ✓ · `deleteBudgetFn` loggea ✓ · tabla no tiene `onDelete: 'cascade'` ✓

#### P2-3: Aritmética de meses incorrecta — `Jan 31 + 1 month = ???`

**Problema**: Calcular manualmente `nextDate` sumando meses con `date.setMonth(date.getMonth() + 1)` produce resultados erróneos en fechas finales de mes. `Jan 31 + 1 month` → `March 3` (febrero no tiene 31 días → overflow a marzo). La regla mensual del día 31 procesa el 3 de marzo en lugar del 28 de febrero.

**Mitigación**: Usar `date-fns` que maneja correctamente los edge cases:

```ts
import { addMonths, addWeeks, addQuarters, addYears } from 'date-fns'

// model/period-utils.ts
export function computeNextDate(rule: BudgetRecurrenceRule, from: Date = new Date()): Date {
  switch (rule.frequency) {
    case 'daily':
      return addDays(from, rule.interval)
    case 'weekly':
      return addWeeks(from, rule.interval)
    case 'monthly':
      return addMonths(from, rule.interval) // Jan 31 + 1 = Feb 28 ✓
    case 'quarterly':
      return addQuarters(from, rule.interval)
    case 'semiannual':
      return addMonths(from, rule.interval * 6)
    case 'yearly':
      return addYears(from, rule.interval)
    default:
      throw new Error(`Unknown frequency: ${rule.frequency}`)
  }
}
```

**Nota**: `date-fns` ya es una dependencia de proyectos TanStack Start; verificar si ya está instalada antes de añadir.

**Checklist**: `date-fns` en `package.json` ✓ · `computeNextDate` usa `addMonths` de date-fns ✓ · test: `Jan 31 + 1 month = Feb 28` ✓ · test: `Feb 28 + 1 month = Mar 28` ✓

#### P2-4: Integer overflow en agregaciones de analytics

**Problema**: La columna `amount` en transactions es `integer` (PostgreSQL signed 32-bit, máx 2,147,483,647 ≈ $21.4M en cents). Una query `SUM(amount)` sobre un budget corporativo con muchas transacciones puede retornar overflow silencioso o error de PostgreSQL.

**Mitigación**:

```ts
// Usar sql<bigint> para todas las agregaciones de SUM:
const [{ totalSpent }] = await db
  .select({
    totalSpent: sql<bigint>`SUM(${transactions.amount})::bigint`,
  })
  .from(transactions)
  .where(eq(transactions.budgetId, budgetId))

// Convertir a number para UI (BigInt no es serializable por JSON):
const totalSpentNumber = Number(totalSpent ?? 0n)

// Alternativa a largo plazo: cambiar columna a bigint en schema
amount: bigint('amount', { mode: 'number' }).notNull()
```

**Checklist**: Todas las queries con `SUM(amount)` usan `::bigint` cast ✓ · resultado BigInt se convierte a Number antes de serializar ✓
